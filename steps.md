# План поэтапной разработки Mockge

Данный документ разбивает техническую спецификацию проекта Mockge на небольшие, независимые задачи. Каждый этап должен завершаться проверкой (верификацией) перед переходом к следующему. Это позволяет минимизировать риски и быстро находить ошибки.

## 📋 Общие принципы работы

- **Итеративность:** Не пытайтесь сделать всё сразу. Завершите задачу → Проверьте → Закоммитьте → Следующая задача.
- **Верификация:** После каждой задачи выполняйте чек-лист проверки (терминал, логи, UI).
- **Ветвление:** Каждую крупную фазу начинайте с новой git-ветки (например, `feat/backend-auth`).

## Этап 1: Инфраструктура и окружение

**Цель:** Поднять локальное окружение для разработки (БД, Кэш, репозитории).

### Задача 1.1: Инициализация репозиториев

**Описание:** Создать структуру проекта (монорепо или мультирепо).

**Действия:**
1. Создать папку `mockge`.
2. Внутри создать папки: `frontend`, `backend`, `proxy`.
3. Инициализировать git в корне и подпапках.
4. Создать `.gitignore` для Java, Node, React.

**✅ Проверка:**
- `git status` показывает чистую структуру.
- Папки созданы и пусты (кроме конфигов).

### Задача 1.2: Docker Compose для локальной разработки

**Описание:** Поднять PostgreSQL и Redis локально.

**Действия:**
1. Создать `docker-compose.yml` в корне.
2. Добавить сервис `postgres` (image: `postgres:15`).
3. Добавить сервис `redis` (image: `redis:7`).
4. Настроить сети и volumes.

**✅ Проверка:**
- Выполнить `docker-compose up -d`.
- Выполнить `docker-compose ps` (статус должен быть **Up**).
- Подключиться к PG: `docker exec -it <container_id> psql -U postgres`.
- Подключиться к Redis: `docker exec -it <container_id> redis-cli`.

## ⚙️ Этап 2: Backend API (Java Spring Boot)

**Цель:** Реализовать управление пользователями и проектами.

### Задача 2.1: Скелет Spring Boot приложения

**Описание:** Базовая настройка backend.

**Действия:**
1. Инициализировать проект Spring Boot (Java 17) через Spring Initializr.
2. Добавить зависимости: Web, Data JPA, PostgreSQL, Lombok, Validation.
3. Настроить `application.properties` (подключение к локальному PG из Docker).

**✅ Проверка:**
- Запустить `mvn spring-boot:run`.
- В логах нет ошибок подключения к БД.
- `curl http://localhost:8080/actuator/health` возвращает **UP**.

### Задача 2.2: Модель данных и миграции (Liquibase)

**Описание:** Создание таблиц Users и Projects.

**Действия:**
1. Добавить зависимость Liquibase.
2. Создать `changelog db/changelog/001-create-users.xml`.
3. Создать `changelog db/changelog/002-create-projects.xml`.
4. Реализовать Entity классы в Java.

**✅ Проверка:**
- Перезапустить приложение.
- Проверить БД: `SELECT * FROM users;` (таблица существует).
- Проверить БД: `SELECT * FROM projects;` (таблица существует).

### Задача 2.3: Аутентификация (JWT)

**Описание:** Регистрация и вход.

**Действия:**
1. Добавить Spring Security.
2. Реализовать `/api/auth/register` и `/api/auth/login`.
3. Настроить генерацию JWT токенов.

**✅ Проверка:**
- `POST /api/auth/register` → статус **201**.
- `POST /api/auth/login` → статус **200**, в теле есть `token`.
- Запрос с неверным паролем → статус **401**.

### Задача 2.4: CRUD Проектов

**Описание:** Управление проектами пользователя.

**Действия:**
1. Реализовать `/api/projects` (GET, POST).
2. Добавить проверку авторизации (только свой пользователь).
3. Реализовать удаление проекта.

**✅ Проверка:**
- С токеном: `GET /api/projects` → пустой список.
- С токеном: `POST /api/projects` → создан проект.
- Без токена: `GET /api/projects` → статус **403**.

## 🚀 Этап 3: Proxy Server (Node.js)

**Цель:** Создать ядро системы, которое обрабатывает запросы к мокам.

### Задача 3.1: Базовый сервер Fastify

**Описание:** Поднять HTTP сервер.

**Действия:**
1. Инициализировать `npm init` в папке `proxy`.
2. Установить `fastify`, `typescript`, `ts-node`.
3. Создать базовый сервер на порту 3000.

**✅ Проверка:**
- `npm run dev`.
- `curl http://localhost:3000` → ответ **"Mockge Proxy"**.
- В терминале видны логи запросов.

### Задача 3.2: Подключение к Redis

**Описание:** Прокси должен читать схемы из Redis.

**Действия:**
1. Установить `ioredis`.
2. Настроить подключение к локальному Redis (из Docker).
3. Создать тестовый ключ `mock:test:schema` в Redis вручную.

**✅ Проверка:**
- В логах прокси: **"Connected to Redis"**.
- Написать тестовый скрипт, который читает ключ из Redis через прокси.

### Задача 3.3: Wildcard Routing (Subdomain parsing)

**Описание:** Определение проекта по поддомену.

**Действия:**
1. Настроить хосты локально (`127.0.0.1 project1.mockge.local`).
2. Реализовать middleware для парсинга `project1` из `host`.
3. Если поддомен не найден в Redis → **404**.

**✅ Проверка:**
- `curl http://project1.mockge.local:3000` → **404** (так как схемы нет).
- В логах видно: **Subdomain parsed: project1**.

## 🔗 Этап 6: Интеграция и Деплой моков

**Цель:** Связать Backend, Frontend и Proxy.

### Задача 6.1: Публикация схемы в Redis

**Описание:** Backend должен уведомлять Proxy об изменениях.

**Действия:**
1. При активации схемы в Backend → записать JSON в Redis (`mock:{subdomain}:schema`).
2. Реализовать эндпоинт `/api/schemas/{id}/deploy`.

**✅ Проверка:**
- Нажать **"Deploy"** в UI.
- Проверить Redis CLI: `GET mock:project1:schema` → возвращает JSON схемы.

### Задача 6.2: Генерация данных в Proxy

**Описание:** Proxy возвращает данные по схеме.

**Действия:**
1. Установить `@faker-js/faker` в Proxy.
2. Реализовать логику: если в схеме поле `email` → генерировать `email`.
3. Обработать GET запрос `/{entity}`.

**✅ Проверка:**
- `curl http://project1.mockge.local:3000/users`.
- В ответе JSON массив с пользователями (`email`, `name`).
- Данные разные при каждом запросе.

### Задача 6.3: Stateful Mocks (Сохранение состояния)

**Описание:** POST запрос сохраняет данные.

**Действия:**
1. Обработать `POST /{entity}` в Proxy.
2. Сохранять созданный объект в Redis (`mock:{subdomain}:state:{entity}`).
3. GET запрос должен возвращать сохраненные объекты.

**✅ Проверка:**
- `POST { "name": "Alex" }` → статус **201**.
- `GET /users` → в списке есть **"Alex"**.

## 🛡️ Этап 8: Безопасность и Полировка

**Цель:** Подготовка к продакшену.

### Задача 8.1: Rate Limiting

**Описание:** Защита от DDOS на уровне Proxy.

**Действия:**
1. Внедрить `fastify-rate-limit`.
2. Настроить лимиты в Redis.

**✅ Проверка:**
- Сделать 100 запросов в секунду.
- 101-й запрос возвращает **429 Too Many Requests**.

### Задача 8.2: Логирование и Мониторинг

**Описание:** Сбор логов.

**Действия:**
1. Настроить Pino в Proxy и Logback в Backend.
2. Вывод логов в JSON формат.

**✅ Проверка:**
- Логи содержат `traceId`.
- Ошибки стека не утекают в ответ клиенту (только в логи).
